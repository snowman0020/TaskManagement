# Design — Phase 2.5: In-app notifications

**Date:** 2026-06-26
**Status:** Approved

Notify users when a task is moved, a task is assigned to them, or a sprint is completed.
Phase 3 (multi-board) will later scope the sprint-complete broadcast to board members.

## Data

A `notifications` collection, one document per recipient per event:
```json
{ "user_id": "<recipient>", "type": "move|assign|sprint_complete",
  "message": "...", "task_id": "<id>"?, "sprint_id": "<id>"?,
  "read": false, "created_at": "<iso utc>" }
```
Indexes: `(user_id, created_at)` and `(user_id, read)`.

## Generation

A helper `app/services/notifications.py::notify(user_ids, type, message, actor_id, task_id?, sprint_id?)`
inserts one doc per recipient, **skipping the actor** and de-duplicating ids.

Hook points (the actor is the authenticated user on each call):
- **move** — `move_task`, `reorder_tasks`, and a status-changing `update_task`: notify the
  task's assignee (the effective assignee after an edit). Message:
  `"<task_number> moved to <status name> by <actor>"`.
- **assign** — `update_task` when `assignee_id` changes to a new user, and `create_task`
  when a task is created already assigned: notify the new assignee. Message:
  `"<actor> assigned <task_number> to you"`.
- **sprint_complete** — `complete_sprint`: notify all **active** users. Message:
  `"<actor> completed sprint <name>"`.

In all cases the actor is never notified about their own action.

## API — new router `app/routers/notifications.py`, prefix `/api/notifications`
- `GET ""` (auth `get_current_user`): `{ items: [...newest first, max 50], unread: <count> }`
  for the current user.
- `POST "/read-all"` (auth `get_current_user`): mark all of the user's notifications read;
  returns the updated count.

## Frontend — `components/NotificationBell.vue` in the top bar (`AppLayout`)
- A 🔔 button with an unread-count badge.
- Fetches on mount; clicking opens a dropdown listing recent notifications (message +
  UTC+7 time via `formatDateTime`), and calls `read-all` so the badge clears.
- Unread items are visually distinct from read ones.

## Error Handling / Edge Cases
- No assignee on a moved task → no notification.
- Actor == recipient (e.g. moving your own task) → skipped.
- Notification insert is best-effort and must not fail the underlying action (move/assign/
  complete still succeed even if notifying errors).

## Testing
- Backend (`http_test.py`): user A moves a task assigned to B → B has a `move`
  notification; A assigns a task to B → B has an `assign` notification; A completes a
  sprint → another user has a `sprint_complete` notification while A does not; `GET`
  returns them with an unread count; `read-all` clears unread.

## Acceptance Criteria
- [ ] Moving a task notifies its assignee; assigning notifies the new assignee; completing a sprint notifies active users.
- [ ] The actor is never notified about their own action.
- [ ] The bell shows an unread count and lists notifications; opening it marks them read.
